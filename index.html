알고리즘 문제 유형 가이드

알고리즘이 어려운 이유는 머리가 나빠서가 아니라, 어떤 유형을 공부해야 하는지와 문제에서 무엇을 봐야 하는지가 처음에는 잘 보이지 않기 때문이다. 이 문서는 프로그래머스 고득점 Kit 유형을 기준으로, 코딩테스트에서 자주 만나는 문제 유형을 설명 위주로 정리한다.

왜 알고리즘이 어렵게 느껴질까

많은 사람이 막히는 지점은 문법보다 문제를 유형으로 분류하는 과정이다.

문제 지문이 길다

입력, 조건, 예외 상황이 섞여 있어서 무엇을 저장하고 무엇을 계산해야 하는지 분리하기 어렵다.

유형 판단이 어렵다

배열로 풀 문제인지, Map이 필요한지, BFS인지 DP인지 판단하는 기준이 아직 익숙하지 않다.

시간복잡도가 낯설다

모든 경우를 봐도 되는지, 정렬이나 이분탐색으로 줄여야 하는지 문제 크기에서 판단해야 한다.

처음 공부할 순서

프로그래머스 유형을 풀기 전에 구현 기초를 먼저 잡고, 이후 자료구조와 탐색으로 넘어간다.

1. 구현 기초

  1. 입력값을 변수, 배열, 2차원 배열에 정확히 저장한다.
  2. 반복문으로 배열을 순회하고 인덱스 범위를 실수하지 않는다.
  3. 문제의 조건을 순서대로 코드 흐름으로 옮긴다.
  4. 상하좌우 이동, 8방향 탐색, 문자열 처리처럼 자주 나오는 구현 패턴을 익힌다.

2. 자료구조

  1. 값을 빠르게 찾을 때는 HashMap과 HashSet을 떠올린다.
  2. 최근 값부터 처리하면 Stack, 먼저 들어온 값부터 처리하면 Queue를 떠올린다.
  3. 가장 작은 값이나 큰 값을 계속 꺼내야 하면 PriorityQueue를 떠올린다.
  4. 문제에서 요구하는 처리 순서가 자료구조 선택의 기준이 된다.

3. 탐색과 완전탐색

  1. 가능한 경우가 적으면 완전탐색으로 모든 후보를 확인한다.
  2. 연결된 영역이나 경로를 찾으면 DFS 또는 BFS를 생각한다.
  3. 최단 거리나 최소 이동 횟수는 BFS가 자주 쓰인다.
  4. 재귀를 쓸 때는 종료 조건과 중복 방문 방지가 핵심이다.

4. 최적화 유형

  1. 정렬하면 선택 기준이 단순해지는지 확인한다.
  2. 정답 후보가 숫자 범위라면 이분탐색을 생각한다.
  3. 현재 최선의 선택이 전체 최선으로 이어지면 탐욕법을 생각한다.
  4. 같은 계산이 반복되면 동적계획법을 생각한다.

기본 문제 유형

프로그래머스 고득점 Kit로 넘어가기 전에 먼저 익숙해져야 하는 입문 유형이다. 각 카드의 문제 버튼은 프로그래머스 입문/Level 1 문제로 연결된다.

코딩테스트 고득점 Kit에 있는 대표 유형을 기준으로 정리했다. 각 유형 설명 아래의 문제 버튼을 누르면 프로그래머스 문제 페이지로 바로 이동한다.

해시

출제 빈도 높음Key-Value빈도수중복 확인

  • 두 목록을 비교해서 빠진 값이나 겹치는 값을 찾는 문제
  • 이름, 번호, 문자열 같은 값을 key로 저장하고 빠르게 찾는 문제
  • 같은 값이 몇 번 등장했는지 세는 문제
  • 문자열이나 객체를 조건에 따라 그룹으로 묶는 문제
  • 배열로 매번 검색하면 느려지는 포함 여부 확인 문제

  • 가장 최근에 들어온 값을 먼저 처리해야 하는 문제
  • 먼저 들어온 작업을 먼저 처리해야 하는 문제
  • 괄호, 짝 맞추기, 이전 상태 되돌리기처럼 최근 상태를 확인하는 문제
  • 작업 대기열, 기능 배포, 프린터 순서처럼 순서대로 처리하는 문제
  • 현재 값과 이전 값을 비교하면서 제거하거나 유지하는 문제

  • 값을 섞거나 합치면서 기준치를 넘겨야 하는 문제

완전탐색

  • 모든 숫자 조합이나 문자열 조합을 만들어 확인하는 문제
  • 선택 가능한 후보가 많지 않아 전부 검사할 수 있는 문제
  • 최소값이나 최대값을 모든 후보 중에서 고르는 문제
  • 조건을 만족하는 후보를 직접 생성해야 하는 문제
  • 재귀, 중첩 반복문, 방문 배열을 활용하는 문제

  • 가장 작은 값, 가장 큰 값, 가장 빠른 값부터 선택하는 문제
  • 정렬한 뒤 앞에서부터 선택하면 최적해가 되는 문제
  • 한 번 선택한 결과를 다시 바꾸지 않는 문제
  • 최소 비용, 최대 개수, 최소 이동 횟수를 구하는 문제
  • 선택 기준이 항상 맞는지 반례를 생각해야 하는 문제

그래프

NodeEdge연결 관계

정점과 간선의 관계를 저장하고, 연결 여부나 이동 비용을 계산하는 유형이다.

  • 서로 연결된 노드들을 탐색하는 문제
  • 네트워크 개수나 연결 그룹을 구하는 문제
  • 간선 정보를 인접 행렬이나 인접 리스트로 바꾸는 문제
  • 방향 그래프와 무방향 그래프를 구분해야 하는 문제
  • 최단 경로, 최소 비용, 연결 가능성을 판단하는 문제
가장 먼 노드순위방의 개수

문제를 읽을 때 볼 것

유형을 맞히는 것보다, 문제 조건에서 필요한 자료구조와 알고리즘을 끌어내는 연습이 중요하다.

입력 크기

  • 크기가 작으면 완전탐색을 먼저 의심한다.
  • 크기가 크면 정렬, 해시, 이분탐색, DP처럼 계산을 줄이는 방법을 생각한다.
  • 2차원 지도라면 좌표 이동과 방문 체크가 필요한지 본다.

반복되는 말

  • 가장 먼저, 가장 최근, 우선순위 같은 표현은 자료구조 힌트가 된다.
  • 최소, 최대, 경우의 수는 DP나 탐욕법으로 이어질 수 있다.
  • 연결, 이동, 네트워크, 경로는 DFS/BFS나 그래프 힌트가 된다.

정답 후보

  • 정답 후보를 모두 만들 수 있으면 완전탐색을 고려한다.
  • 정답 후보가 숫자 범위라면 이분탐색을 고려한다.
  • 이전 후보의 결과가 다음 후보에 재사용되면 DP를 고려한다.

선택의 영향

  • 현재 선택이 나중 선택에 영향을 주지 않으면 탐욕법 가능성이 있다.
  • 현재 선택이 이후 상태를 바꾸면 탐색이나 DP가 필요할 수 있다.
  • 한 번 방문한 상태를 다시 계산한다면 저장할 상태를 찾아야 한다.

다음 방향

처음에는 모든 유형을 한 번에 잡으려 하지 말고, 구현 기초와 자주 나오는 유형부터 넓혀 가는 편이 좋다.

추천 순서: 입출력·조건문·반복문 → 배열·문자열 → 2차원 배열·시뮬레이션 → 해시 → 스택·큐 → 정렬 → 완전탐색 → DFS/BFS → 힙 → 이분탐색 → 탐욕법 → DP → 그래프
이 페이지는 알고리즘 유형 설명과 프로그래머스 대표 문제 링크를 연결한 학습 안내 자료다.
